# CloudSafe HelpDesk

Тикет-система на Flask + SQLite для внутренней команды поддержки.

## Что реализовано

- Пользователи системы: только `admin` и `operator`.
- Организации (`Organization`) с реквизитами и контактами.
- Сотрудники организаций (`Employee`) как отдельная сущность (без входа в систему).
- Задачи:
  - обязательная привязка к организации,
  - опциональная привязка к сотруднику,
  - статусы, приоритет, ответственный оператор,
  - комментарии,
  - архив/восстановление,
  - история изменений полей и история статусов.
- Разграничение доступа операторов по организациям (`OperatorOrganizationAccess`).
- Уведомления о новых задачах/комментариях:
  - Telegram (если есть chat id),
  - иначе email.
- API создания задач по Bearer-токену организации.
- Отчеты для администратора (`/reports`) + экспорт CSV.
- Системные настройки (`/settings/system`):
  - название сайта,
  - логотип,
  - favicon,
  - основные цвета интерфейса,
  - сброс к дефолту.
- Справочник приоритетов (`/priorities`, фиксированные значения low/medium/high).

## API

### POST /api/tasks

Создание задачи по токену организации.

Аутентификация:

- `Authorization: Bearer <ORG_TOKEN>`

Тело JSON:

```json
{
  "theme": "Не работает VPN",
  "content": "С утра нет подключения",
  "due_date": "2026-12-31",
  "priority": "high",
  "employee_id": 12,
  "assigned_to_id": 5
}
```

Пояснения:

- `organization_id` можно не передавать: он определяется токеном.
- Если `organization_id` передан, он должен совпадать с организацией токена.
- `employee_id` опционален, но должен принадлежать организации токена.
- `assigned_to_id` опционален: оператор должен иметь доступ к этой организации.

Ответ: `201` с данными созданной задачи.

## Управление API-токеном организации

- В карточке организации (`/organizations/<id>/edit`) есть кнопка генерации нового токена.
- В базе хранится только hash токена (`api_token`) и префикс (`api_token_prefix`).
- Полное значение токена показывается только один раз после генерации.

## Отчеты

Страница: `/reports` (только администратор).

Доступно:

- период в днях или диапазон дат,
- всего создано задач за период,
- выполнено (по статусам с `is_final = true`),
- не выполнено,
- детализация по каждому статусу,
- экспорт CSV.

## Миграция со старой версии

Перед обновлением рекомендуется сделать backup файла БД (`instance/ticket_system.db`).

При запуске новой версии автоматически выполняется простая миграция:

1. Добавляются недостающие колонки в существующие таблицы.
2. Старые пользователи с ролью `client` конвертируются в `Employee`:
   - создается/используется организация,
   - задачи переносятся на `employee_id` + `organization_id`.
3. Старые client-аккаунты деактивируются и переводятся в техническую роль `legacy_client`.
4. Старые связи доступа `client_access` конвертируются в `operator_organization_access`.
5. Для задач без организации проставляется организация по умолчанию.

Если после миграции нужно ручное выравнивание данных:

- проверьте карточки организаций,
- проверьте сотрудников,
- проверьте доступы операторов (`/access`).

## Ограничения и заметки

- Архивные задачи нельзя редактировать, менять статус и комментировать.
- Удаление организации запрещено, если есть связанные задачи или сотрудники.
- Удаление сотрудника запрещено, если есть связанные задачи.
- Оператор видит только организации, к которым у него выдан доступ.
