# Предложенные задачи по улучшению кодовой базы

## 1) Опечатка (UI/локализация)
**Проблема:** в русскоязычном интерфейсе встречается английская метка `active`, что выглядит как невыправленная строка/опечатка локализации.

**Где видно:**
- `templates/users.html` — badge со значением `active`.
- `templates/employees.html` — badge со значением `active`.

**Задача:** заменить `active` на локализованную подпись (`Активен` / `Активный`) и унифицировать стиль меток активности во всех шаблонах.

---

## 2) Исправление ошибки (runtime/совместимость модели и кода)
**Проблема:** бизнес-логика все еще использует строковый `priority`, но модель `Task` уже перешла на `priority_id` и связь `priority`.

**Признаки рассинхронизации:**
- В `app.py` добавляется колонка `tasks.priority` в миграции и далее идет фильтрация/присвоение `Task.priority`.
- В `models.py` у `Task` отсутствует поле `priority` и вместо него есть `priority_id`.
- В `services.py` используется `Priority.ALL` и `Priority.LABELS`, но в текущей модели `Priority` таких атрибутов нет.

**Риск:** падения при создании/редактировании/фильтрации задач и при открытии страниц, где вызываются `priority_choices()`.

**Задача:** привести код к единой модели приоритетов (через таблицу `priorities`), убрать устаревшие обращения к `Task.priority`, пересобрать валидацию/формы/API под `priority_id` (или выбрать обратный путь и откатить модель, но последовательно во всех слоях).

---

## 3) Исправление комментария/документации (несоответствие)
**Проблема:** README заявляет, что приоритеты фиксированные (`low/medium/high`), но в модели есть полноценная таблица `priorities`, а в приложении есть экран `/priorities` для управления справочником.

**Где видно:**
- `README.md`: указано «фиксированные значения low/medium/high».
- `models.py`: сущность `Priority` как отдельная таблица.
- `app.py`: маршрут `/priorities` и CRUD по статусам/приоритетам в интерфейсе.

**Задача:** обновить README и описание API под фактическую модель приоритетов (справочник в БД), включая примеры payload/фильтров и миграционных замечаний.

---

## 4) Улучшение теста (покрытие критического сценария)
**Проблема:** в репозитории отсутствует тестовый набор для ключевых сценариев валидации и API.

**Задача:** добавить интеграционный тест для `POST /api/tasks`, который проверяет:
1. валидный Bearer-токен организации;
2. создание задачи с корректным приоритетом и сотрудником своей организации;
3. отказ при передаче `organization_id`, не совпадающего с организацией токена;
4. отказ при назначении оператора без доступа к организации.

**Ожидаемый эффект:** раннее обнаружение регрессий в авторизации и связях `organization/employee/assignee`.
